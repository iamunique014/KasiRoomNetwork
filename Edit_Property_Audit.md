# Edit Property Audit

## 1. Executive Summary

The current **Edit Property** feature is partially implemented for editing the core `Property` row and its related `Address` row. The implemented path is:

```text
My Properties â†’ View Property / Property Details â†’ Edit â†’ Edit Property form â†’ Save Changes â†’ Property Details
```

The feature has a working `EditProperty` GET/POST pair, a dedicated `EditPropertyViewModel`, repository methods, and stored procedures for loading and saving editable property/address fields. The strongest part of the implementation is the database-level ownership filter in `sp_Property_Get_For_Edit` and `sp_Property_Update`, which uses `PropertyId + LandlordId`.

However, the feature is not complete as a full property management experience:

- The `EditProperty` controller actions are **not decorated with `[Authorize(Roles = "Landlord")]`**, even though they rely on the logged-in user claim.
- The edit form has **no Cancel/Back action**.
- The edit form has **minimal validation** because `EditPropertyViewModel` has no validation attributes.
- Amenities and photos are visible around the property flow, but **Edit Amenities** and **Edit Photos** are not implemented as complete edit experiences.
- The `PropertyDetails` page contains a link to `EditPropertyAmenities`, but no matching controller action or view exists.
- `PropertyDetails` is publicly reachable and can show landlord contact data for any property id; this may be intentional for tenant-facing viewing, but it is not yet clearly separated from a landlord-only management hub.
- Photo loading appears incomplete on `PropertyDetails`: the view renders `Model.Photos`, but `PropertyController.PropertyDetails` only loads amenities and does not call `GetPropertyPhotos`.
- `MyProperties.cshtml` has route mismatches: the empty-state Create button points to `Property/Create`, while the controller action is `CreateProperty`; the Delete form posts to `Delete`, while the controller action is `DeleteProperty`.

Overall assessment: **Edit Property core fields are partially implemented; full property management hub alignment is early-stage.**

## 2. UX Flow Audit

### Current UX Flow

```text
Step 1: Landlord opens My Properties.
Route: GET /Property/MyProperties
Controller: PropertyController.MyProperties()
View: Views/Property/MyProperties.cshtml

Step 2: My Properties displays property cards.
Each card has Add Listing, View Property, and Delete buttons.
The direct Edit Property entry point is not on this page.

Step 3: Landlord clicks View Property.
Route generated by tag helpers: GET /Property/PropertyDetails?propertyId={propertyId}
Controller: PropertyController.PropertyDetails(int propertyId)
View: Views/Property/PropertyDetails.cshtml

Step 4: Property Details computes canEditProperty in the view.
Condition: current user is in role Landlord AND current user id equals Model.LandlordUserId.
If true, it displays an Edit button in the Property Info card.

Step 5: Landlord clicks Edit.
Route generated by tag helpers: GET /Property/EditProperty?propertyId={propertyId}
Controller: PropertyController.EditProperty(int propertyId)
Repository: IPropertyRepository.GetPropertyForEditAsync(propertyId, landlordUserId)
Stored procedure: sp_Property_Get_For_Edit
ViewModel: EditPropertyViewModel
View: Views/Property/EditProperty.cshtml

Step 6: Landlord edits fields.
Fields: PropertyName, PropertyType, TotalRooms, Province, City, Suburb, Street.

Step 7: Landlord clicks Save Changes.
Route generated by form tag helper: POST /Property/EditProperty
Controller: PropertyController.EditProperty(EditPropertyViewModel model)
Validation: ModelState.IsValid only; model has no validation attributes.
Repository: IPropertyRepository.UpdatePropertyAsync(model, landlordUserId)
Stored procedure: sp_Property_Update

Step 8: On successful save, controller sets TempData["Success"] = "Property updated successfully." and redirects to PropertyDetails.
Redirect: GET /Property/PropertyDetails?propertyId={model.PropertyId}

Step 9: Property Details renders again.
Current view does not render TempData["Success"], so the success message is set but not visible on that page.
```

### Entry Points

- **Primary discovered entry point:** `Views/Property/PropertyDetails.cshtml` shows an `Edit` link when the authenticated landlord owns the property.
- **Indirect entry point:** `Views/Property/MyProperties.cshtml` links to `PropertyDetails`, but does not directly expose Edit Property.
- **No navbar or dashboard Edit Property entry point was found in the audited files.**

### Buttons or Links Found

#### My Properties

- `+ Add Property` â†’ `Property/CreateProperty`.
- Empty-state `Create Property` â†’ `Property/Create` (**route mismatch; no `Create` action found in `PropertyController`**).
- `Add Listing` â†’ `Listing/CreateListing?propertyId={propertyId}`.
- `View Property` â†’ `Property/PropertyDetails?propertyId={propertyId}`.
- `Delete` form â†’ posts to `Delete` (**route mismatch; controller action is `DeleteProperty`**).

#### Property Details

- `Edit` in the property information card â†’ `Property/EditProperty?propertyId={propertyId}`.
- `Edit` in the amenities card â†’ `Property/EditPropertyAmenities?propertyId={propertyId}` (**no matching action/view found**).

#### Edit Property

- `Save Changes` submit button only.
- No Cancel button.
- No Back button.
- No separate Save and Continue / Save Draft action.

### Routes Used

The project uses conventional MVC routing through controller/action tag helpers. Relevant routes are:

- `GET /Property/MyProperties`
- `GET /Property/PropertyDetails?propertyId={id}`
- `GET /Property/EditProperty?propertyId={id}`
- `POST /Property/EditProperty`
- `GET /Listing/CreateListing?propertyId={id}`
- `POST /Property/DeleteProperty` exists in controller, but the My Properties view posts to `Delete`, not `DeleteProperty`.

### What Happens After Save?

- POST `EditProperty` checks `ModelState.IsValid`.
- It reads `ClaimTypes.NameIdentifier`.
- It calls `UpdatePropertyAsync(model, landlordUserId)`.
- It sets `TempData["Success"] = "Property updated successfully."`.
- It redirects to `PropertyDetails` with the edited `PropertyId`.

### What Happens After Cancel?

Cancel is not implemented on the `EditProperty` page.

### Success/Error Messages

Implemented messages related to this flow:

- `TempData["Success"] = "Property updated successfully."` after Edit Property save.
- No rendering of `TempData["Success"]` was found in `PropertyDetails.cshtml` or `EditProperty.cshtml`.
- Edit form renders `asp-validation-for="PropertyName"` only; other fields do not render validation spans.
- The form does not include `asp-validation-summary`.
- Since the view model lacks validation attributes, most server/client validation messages are not defined.
- `AddPropertyPhotos` has upload validation messages for missing photo, unsupported type, and size above 2MB, plus `TempData["PhotoUploaded"]` rendering on the photo upload view.

### UX Gaps

- No Cancel/Back action on Edit Property.
- Success TempData is set but not displayed on return to Property Details.
- No error handling message if `sp_Property_Update` updates zero rows because of ownership mismatch or inactive property.
- No direct Edit button on My Properties cards.
- Property Details has an amenities edit link to a missing action.
- No implemented Edit Photos entry point from Property Details.
- Property Details does not visibly provide Listings Management; My Properties has Add Listing, but the intended hub should own this flow.
- Route mismatches in My Properties can break empty-state Create and Delete.

### Controller

```text

```

### GET Actions

#### `CreateProperty()`

- Attributes: `[Authorize(Roles = "Landlord")]`, `[HttpGet]`.
- Route: `GET /Property/CreateProperty`.
- Purpose: Creates initial property form and loads amenities.
- Relevance: Upstream creation flow before edit exists.

#### `AddPropertyPhotos(int propertyId)`

- Attributes: `[Authorize(Roles = "Landlord")]`, `[HttpGet]`.
- Route: `GET /Property/AddPropertyPhotos?propertyId={id}`.
- Purpose: Upload property photos after create.
- Relevance: Photo add exists, but edit/manage photos does not.
- Ownership check: none in controller; repository count method also receives only `propertyId`.

#### `MyProperties()`

- Attributes: `[Authorize(Roles = "Landlord")]`, `[HttpGet]`.
- Route: `GET /Property/MyProperties`.
- Purpose: Lists properties for the logged-in landlord.
- Ownership check: passes current user id to `GetAllPropertiesByLandlord`.

#### `PropertyDetails(int propertyId)`

- Attributes: `[HttpGet]`; no `[Authorize]`.
- Route: `GET /Property/PropertyDetails?propertyId={id}`.
- Purpose: Displays property details and loads amenities.
- Ownership check: none in controller; the view hides owner actions using `canEditProperty`.
- Security note: Public access may be acceptable for tenant-facing details, but management-hub behavior should be separated or protected.

#### `EditProperty(int propertyId)`

- Attributes: `[HttpGet]`; **no `[Authorize]`**.
- Route: `GET /Property/EditProperty?propertyId={id}`.
- Purpose: Loads edit form for owned property.
- Ownership check: passes current user id to `GetPropertyForEditAsync`; stored procedure filters by `LandlordId`.
- Unauthenticated behavior: if claim is missing, returns `Challenge()`.

#### `PropertySubmitted(int propertyId)`

- Attributes: none.
- Route: `GET /Property/PropertySubmitted?propertyId={id}`.
- Purpose: Completion page after photo upload.
- Relevance: Not an edit action.

### POST Actions

#### `CreateProperty(CreatePropertyViewModel model)`

- Attributes: `[Authorize(Roles = "Landlord")]`, `[HttpPost]`, `[ValidateAntiForgeryToken]`.
- Route: `POST /Property/CreateProperty`.
- Purpose: Creates property and selected amenities, then redirects to AddPropertyPhotos.
- Validation: uses `ModelState.IsValid`.

#### `AddPropertyPhotos(int propertyId, IFormFile photo, bool isPrimary)`

- Attributes: `[Authorize(Roles = "Landlord")]`, `[HttpPost]`, `[ValidateAntiForgeryToken]`.
- Route: `POST /Property/AddPropertyPhotos`.
- Purpose: Uploads a new property photo.
- Validation: file required, extension `.jpg/.jpeg/.png`, max size 2MB.
- Ownership check: none.

#### `EditProperty(EditPropertyViewModel model)`

- Attributes: `[HttpPost]`, `[ValidateAntiForgeryToken]`; **no `[Authorize]`**.
- Route: `POST /Property/EditProperty`.
- Purpose: Updates property and address fields.
- Validation: `ModelState.IsValid` only.
- Ownership check: passes current user id to repository and stored procedure filters by `LandlordId`.
- Success path: sets `TempData["Success"]`, redirects to `PropertyDetails`.

#### `DeleteProperty(int propertyId)`

- Attributes: `[HttpPost]`, `[ValidateAntiForgeryToken]`; **no `[Authorize]`**.
- Route: `POST /Property/DeleteProperty`.
- Purpose: Soft-deletes a property and its listings through stored procedure.
- Ownership check: passes current user id to stored procedure.
- Route mismatch: My Properties view posts to `Delete`, not `DeleteProperty`.

### Validation

- Edit Property uses `ModelState.IsValid`.
- `EditPropertyViewModel` has no validation attributes, so server-side validation is minimal.
- `EditProperty.cshtml` only renders validation output for `PropertyName`.
- Anti-forgery is present on POST actions.

### Authorization and Ownership Checks

- `CreateProperty`, `AddPropertyPhotos`, and `MyProperties` are explicitly landlord-only.
- `EditProperty` GET/POST are not explicitly authorized but do require a logged-in claim at runtime.
- Ownership enforcement for edit is in repository/SP via `LandlordId`.
- `PropertyDetails` is public and uses UI-level hiding for owner actions.
- Add photo methods lack ownership enforcement.

### Missing Controller Logic

- Add `[Authorize(Roles = "Landlord")]` to EditProperty GET/POST and DeleteProperty POST.
- Detect whether update affected zero rows and show a meaningful error/NotFound/Forbid.
- Render or pass success messages consistently.
- Add Cancel route support by providing links in view, not necessarily a controller method.
- Implement EditPropertyAmenities GET/POST or remove the broken link.
- Implement EditPropertyPhotos / ManagePropertyPhotos GET/POST if the intended flow includes Edit Photos.
- Load property photos in `PropertyDetails` or update repository/SP to populate `Photos`.
- Consider making a landlord-only management hub action separate from public `PropertyDetails`.
- Add ownership checks to AddPropertyPhotos and photo count/upload repository methods.

## 9. Security Audit

### Authorization

Current Edit Property implementation relies on authentication state but lacks explicit authorization attributes:

- `EditProperty` GET: no `[Authorize(Roles = "Landlord")]`.
- `EditProperty` POST: no `[Authorize(Roles = "Landlord")]`.
- `DeleteProperty` POST: no `[Authorize(Roles = "Landlord")]`.

The actions still call `User.FindFirstValue(ClaimTypes.NameIdentifier)` and return `Challenge()` if no user id exists, but explicit role-based authorization should be present for landlord-only actions.

### Can a Landlord Edit Another Landlord's Property?

Based on current stored procedures: likely **no** for the core Edit Property fields, because:

- GET edit data calls `sp_Property_Get_For_Edit` with `PropertyId` and current `LandlordId`.
- POST update calls `sp_Property_Update` with `PropertyId` and current `LandlordId`.
- Both stored procedures filter by `LandlordId`.

However, the controller does not explicitly return `Forbid()` or an ownership error. If the update affects zero rows, it still redirects with a success message.

### Ownership Checks by Layer

#### Controller

- Edit GET/POST: obtains current user id but does not independently verify ownership.
- PropertyDetails: view hides edit buttons based on owner match; controller does not enforce owner-only because it is public-style.
- AddPropertyPhotos: no ownership check.

#### Repository

- Edit methods accept `landlordId` and pass it to SPs.
- Photo and amenity methods generally do not accept `landlordId`.

#### Stored Procedure

- Strong: `sp_Property_Get_For_Edit`, `sp_Property_Update`, `sp_Landlord_Get_All_Properties`.
- Missing: photo add/count/get and amenity add/remove/get do not verify ownership.

### Security Risks

- Missing `[Authorize(Roles = "Landlord")]` on edit/delete actions.
- Core update can silently no-op and still show success for unauthorized/invalid property ids.
- Add property photo accepts `propertyId` without ownership verification; any authenticated landlord could potentially upload to another property if they know the id.
- Amenity add/remove methods are not ownership-safe if exposed by future actions without additional checks.
- PropertyDetails public endpoint returns landlord phone and full name for any property id.
- `sp_Property_Get_By_Id` does not filter inactive properties.
- Delete form route mismatch may create confusion and broken behavior.
- No audit logging around property edits.

## 10. Data Integrity Audit

### What Updates Together?

Edit Property save updates:

- `Property.PropertyName`
- `Property.PropertyType`
- `Property.TotalRooms`
- `Address.Province`
- `Address.City`
- `Address.Suburb`
- `Address.Street`
- `Address.UpdatedAt`

These updates happen together in `sp_Property_Update` inside a transaction.

### What Updates Separately?

- Property amenities are handled separately by `sp_PropertyAmenity_Add` and `sp_PropertyAmenity_Remove`, but no edit UI/action exists.
- Property photos are handled separately by `sp_Property_Add_Photo`, `sp_Property_Get_Photos`, and `sp_Property_Get_Photo_Count`.
- Listings are created separately through `ListingController.CreateListing`.

### Transaction Safety

- Core property/address edit uses a transaction and TRY/CATCH.
- Create property uses a transaction and TRY/CATCH.
- Delete property uses a transaction and TRY/CATCH.
- Property photo add does not use a transaction despite potentially updating existing photos and inserting a new photo.
- Amenity add/remove do not use transactions, though each is a simple single operation.

### Partial Update Risks

- `sp_Property_Update` protects Property + Address consistency, but it does not report whether either update affected zero rows.
- `sp_Property_Add_Photo` can unset existing primary photos and fail before insert, leaving no primary photo.
- Future amenity bulk edits could partially apply if implemented as a loop without a transaction.
- Property photo count includes inactive photos, while photo retrieval filters active photos.
- `PropertyDetails` may not show photos because controller does not load them into the model.

### Data Integrity Risks

- No controlled list/check constraint for `PropertyType`.
- `TotalRooms` nullable and not range-checked.
- Edit view model accepts strings without max-length validation matching SQL column sizes.
- `Address.Province` is NOT NULL, but view model has no required validation.
- `Property.LandlordId` is nullable despite representing ownership.
- `PropertyPhoto.IsActive` default is `0`, while insert SP sets it to `1`; direct inserts could create inactive photos unexpectedly.
- No database constraint ensures only one primary photo per property.
